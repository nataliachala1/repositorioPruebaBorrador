# ADR-004: Estrategia de versionamiento con ramas develop, qa y main

## Contexto

El repositorio del proyecto existe y contiene los artefactos relacionados con el modelo de base de datos (DDL, ADR, análisis de dominios). Sin embargo, no se ha definido una estrategia clara de manejo de ramas para controlar el flujo de cambios.

Dado que el desarrollo se realiza en equipo y los cambios pueden afectar componentes críticos del sistema, es necesario establecer un flujo controlado que permita integrar, validar y liberar cambios de manera segura.

## Problema

Sin una estrategia de ramas definida:

* Se pueden realizar cambios directamente sobre `main`, afectando la estabilidad del sistema
* Se mezcla código en desarrollo con código no probado
* No existe un entorno claro para validación (QA)
* Se dificulta la trazabilidad de cambios
* Aumenta el riesgo de errores en producción

Esto genera desorganización en el equipo y compromete la calidad del producto final.

## Decisión

Se adopta una estrategia de versionamiento basada en tres ramas principales:

* `develop`: rama de integración donde los desarrolladores consolidan su trabajo diario.
* `qa`: rama destinada a pruebas y validación antes de la aprobación final.
* `main`: contiene únicamente código estable, validado y listo para producción.

Flujo de trabajo definido:

```id="w3e1az"
feature/* → develop → qa → main
```

Reglas:

* Las ramas `feature/*` se crean a partir de `develop` para desarrollar funcionalidades específicas.
* Los cambios en `feature/*` se integran a `develop` mediante merge.
* Solo cambios estables de `develop` pasan a `qa` para validación.
* Solo cambios aprobados en `qa` se integran a `main`.
* No se permiten commits directos a `main`.
* Los merges deben realizarse de manera controlada para garantizar la calidad del código.

## Justificación técnica

Esta estrategia permite separar claramente los entornos de desarrollo, validación y producción sin la complejidad completa de modelos como GitFlow.

A diferencia de enfoques más simples como trunk-based development, este modelo:

* Proporciona una etapa explícita de validación (`qa`) antes de producción
* Reduce el riesgo de desplegar cambios no probados
* Facilita el control de calidad en entornos académicos o de evaluación

Además, es más sencillo de implementar y mantener que GitFlow completo, ya que reduce la cantidad de ramas permanentes y reglas complejas.

## Consecuencias o impacto esperado

### Positivas

* Se mejora la organización del equipo de desarrollo
* Se reduce el riesgo de errores en producción
* Se establece un flujo claro y controlado de cambios
* Se facilita la validación antes de liberar código
* Se mejora la trazabilidad de los cambios entre ramas

### Negativas / Costos

* Mayor disciplina requerida por parte del equipo
* Incremento en el número de pasos para integrar cambios
* Posibles retrasos si no se gestionan correctamente los merges

### Suposiciones

* Todos los desarrolladores seguirán el flujo definido (`feature → develop → qa → main`)
* No se realizarán cambios directos sobre `main`
* La rama `qa` será utilizada activamente para validación antes de producción
* El equipo aplicará buenas prácticas de commits y merges
