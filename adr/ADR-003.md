# ADR-003: Implementación de Liquibase para el versionamiento del DDL

## Contexto

Actualmente, el modelo de base de datos se encuentra definido en un único archivo `.sql`, el cual contiene la estructura completa del esquema.

Este enfoque no permite llevar un control detallado de los cambios realizados sobre el DDL (Data Definition Language), dificultando la trazabilidad, mantenimiento y evolución del modelo a lo largo del tiempo, especialmente en entornos colaborativos.

## Problema

Cuando múltiples desarrolladores realizan cambios sobre el modelo sin un control de versiones estructurado:

* Se generan conflictos entre versiones del archivo `.sql`
* No existe trazabilidad clara de qué cambios se realizaron, cuándo y por quién
* Es difícil reproducir el estado de la base de datos en diferentes ambientes (desarrollo, QA, producción)
* No se pueden aplicar cambios de forma incremental ni controlada
* Se incrementa el riesgo de errores en despliegues

Esto afecta directamente la calidad, estabilidad y mantenibilidad del sistema.

## Decisión

Se adopta **Liquibase** como herramienta para el versionamiento y gestión de cambios del DDL.

Se define una estructura de changelogs organizada por dominios funcionales, cumpliendo con el criterio de un changelog por dominio:

```
liquibase/
├── changelog-master.xml
└── changelogs/
    ├── geography.xml
    ├── airline.xml
    ├── identity.xml
    ├── security.xml
    ├── customer.xml
    ├── airport.xml
    ├── aircraft.xml
    ├── flight.xml
    ├── reservation.xml
    ├── boarding.xml
    ├── payment.xml
    └── invoicing.xml
```

El archivo `changelog-master.xml` actuará como punto central, incluyendo todos los changelogs de los dominios.

Cada cambio en la base de datos se registrará como un *changeset*, permitiendo su ejecución controlada, ordenada y versionada.

## Justificación técnica

Liquibase permite gestionar los cambios en la base de datos de manera similar a un sistema de control de versiones como Git, lo cual ofrece ventajas frente a alternativas como scripts manuales o herramientas como Flyway:

* Permite definir cambios declarativos en formato XML, SQL o YAML
* Mantiene un historial completo de cambios mediante la tabla `DATABASECHANGELOG`
* Soporta rollback de cambios, lo cual no es posible fácilmente con scripts manuales
* Facilita la ejecución incremental de cambios sin duplicaciones
* Permite organizar los cambios por dominios, alineándose con la estructura del modelo

A diferencia de scripts `.sql` tradicionales, Liquibase garantiza consistencia entre ambientes y evita la ejecución repetida de cambios.

## Consecuencias o impacto esperado

### Positivas

* Se obtiene trazabilidad completa de los cambios en el esquema de la base de datos
* Se mejora la coordinación entre desarrolladores
* Se garantiza la reproducibilidad del modelo en diferentes entornos
* Se facilita el despliegue controlado de cambios
* Se alinea el versionamiento del DDL con prácticas modernas de desarrollo

### Negativas / Costos

* Curva de aprendizaje inicial para el equipo
* Necesidad de definir y mantener los changelogs
* Incremento en la complejidad del proceso de despliegue

### Suposiciones

* Todos los cambios al esquema se realizarán exclusivamente a través de Liquibase
* Cada dominio funcional tendrá su propio changelog
* El equipo seguirá una convención de nombrado y organización de changesets
* Se integrará Liquibase en el flujo de desarrollo y despliegue del sistema
