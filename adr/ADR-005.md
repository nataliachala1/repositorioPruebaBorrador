# ADR-005: Estrategia de datos de prueba para poblamiento del modelo

## Contexto

El modelo de base de datos incluye múltiples dominios funcionales interrelacionados mediante claves foráneas, lo que genera dependencias entre tablas.

Para validar el correcto funcionamiento del modelo, es necesario poblar la base de datos con datos de prueba. Sin embargo, actualmente no existe una estrategia definida para la inserción de estos datos.

## Problema

La inserción de datos sin un orden estructurado puede generar errores de integridad referencial, ya que muchas tablas dependen de otras previamente pobladas.

Por ejemplo:

* No se puede insertar un `flight` sin un `aircraft`
* No se puede registrar un `payment` sin una `sale`
* No se puede crear un `airport` sin una `address`

Esto provoca:

* Fallos en la carga de datos
* Inconsistencias en el modelo
* Dificultad para realizar pruebas funcionales
* Pérdida de tiempo en la corrección de errores

## Decisión

Se define una estrategia de poblamiento de datos de prueba basada en el orden de dependencias entre dominios funcionales.

El orden de inserción será el siguiente:

1. Geografía y datos de referencia
2. Aerolínea
3. Identidad
4. Seguridad
5. Clientes y fidelización
6. Aeropuerto
7. Aeronaves
8. Operaciones de vuelo
9. Ventas y reservas
10. Abordaje
11. Pagos
12. Facturación

Cada dominio contará con su propio script de inserción de datos, permitiendo modularidad y reutilización.

Adicionalmente:

* Los datos deben ser **realistas pero ficticios**, evitando el uso de información sensible
* Los scripts deben respetar las relaciones entre tablas
* Se deben incluir datos suficientes para validar los principales flujos del sistema

## Justificación técnica

El modelo relacional implementado depende fuertemente de la integridad referencial mediante claves foráneas, lo que obliga a respetar un orden lógico de inserción.

Definir una estrategia clara:

* Evita errores por violación de restricciones
* Permite ejecutar los scripts de forma automatizada
* Facilita la validación del modelo en entornos de desarrollo y pruebas
* Se alinea con la organización por dominios utilizada en el diseño del sistema

Además, la separación por dominios permite mantener los datos organizados y escalables.

## Consecuencias o impacto esperado

### Positivas

* Se garantiza la correcta carga de datos respetando las dependencias
* Se facilita la ejecución repetible de pruebas
* Se mejora la mantenibilidad de los scripts de datos
* Se permite validar el comportamiento del sistema de forma integral

### Negativas / Costos

* Requiere definir y mantener múltiples scripts de inserción
* Incrementa el tiempo inicial de configuración
* Necesidad de coordinar cambios cuando se modifica el modelo

### Suposiciones

* Todos los scripts de inserción respetarán el orden definido
* Cada dominio tendrá su propio archivo de datos de prueba
* Los datos serán suficientes para cubrir los casos de uso principales
* Se realizarán validaciones posteriores a la carga (conteo de registros, verificación de relaciones) para asegurar la integridad de los datos
