# ADR-002: Manejo de roles con permisos diferenciados

## Contexto

El modelo de datos ya incluye un conjunto de tablas destinadas a la gestión de seguridad y control de acceso: `security_role`, `security_permission`, `user_account`, `user_role` y `role_permission`.

Estas estructuras permiten definir roles, permisos y la asignación de dichos roles a los usuarios del sistema. Sin embargo, no se ha establecido una política clara sobre cómo se deben definir los roles ni cómo se asignan los permisos dentro del sistema.

Dado que el sistema gestiona información crítica como reservas, pagos, vuelos y datos personales, es necesario garantizar que cada usuario tenga acceso únicamente a las funcionalidades que le corresponden.

## Problema

Si todos los usuarios tienen los mismos permisos o no existe una correcta asignación de roles:

* Se compromete la seguridad del sistema
* Se pueden realizar acciones no autorizadas (ej: modificar pagos, eliminar reservas)
* Se pierde trazabilidad sobre quién ejecuta cada acción
* Se incrementa el riesgo de errores operativos y fraude

Además, sin una estructura clara de roles, la tabla `role_permission` pierde su propósito, generando inconsistencias en la asignación de accesos.

## Decisión

Se define un modelo de control de acceso basado en roles (RBAC), utilizando las tablas existentes.

Se establecen los siguientes roles principales:

* **Administrador**: acceso total al sistema (gestión de usuarios, configuración, auditoría).
* **Agente de ventas**: gestión de reservas, emisión de tiquetes, atención al cliente.
* **Operaciones**: gestión de vuelos, asignación de tripulación, estados de vuelo.
* **Auditor**: acceso de solo lectura a información crítica para revisión y control.

Los permisos se gestionan mediante la tabla `security_permission`, donde cada permiso representa una acción específica (crear, consultar, actualizar, eliminar).

La tabla `role_permission` se utiliza para asignar permisos a cada rol, garantizando que:

* Cada rol tenga únicamente los permisos necesarios
* Se mantenga el principio de mínimo privilegio
* Los accesos sean fácilmente auditables

La asignación de roles a usuarios se realiza mediante la tabla `user_role`.

## Justificación técnica

El modelo ya cuenta con una estructura completa para RBAC, pero su efectividad depende de una correcta definición y uso.

El uso de `role_permission` permite desacoplar roles de permisos, evitando redundancia y facilitando cambios futuros sin afectar directamente a los usuarios.

Además:

* Permite escalar el sistema agregando nuevos roles sin modificar la lógica central
* Facilita el mantenimiento al centralizar los permisos
* Garantiza consistencia en el acceso a funcionalidades críticas

Este enfoque aprovecha completamente las tablas existentes sin necesidad de rediseñar el modelo.

## Consecuencias o impacto esperado

### Positivas

* Se mejora la seguridad del sistema mediante control de acceso granular
* Se garantiza que cada usuario acceda solo a lo que le corresponde
* Se facilita la auditoría de acciones realizadas por los usuarios
* Se mantiene la flexibilidad para agregar nuevos roles o permisos

### Negativas / Costos

* Mayor esfuerzo inicial para definir correctamente roles y permisos
* Incremento en la complejidad de la gestión de usuarios
* Necesidad de mantener actualizada la matriz de permisos

### Suposiciones

* Cada usuario puede tener uno o varios roles asignados
* Los permisos están correctamente definidos y normalizados en `security_permission`
* El sistema aplicará validaciones de acceso basadas en roles en todas las operaciones críticas
